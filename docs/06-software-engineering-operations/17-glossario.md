## Terminologia Essencial

### Conceitos de Confiabilidade

**Toil** Trabalho manual, repetitivo, automatizável, que não agrega valor
diferenciado e que cresce linearmente com o sistema. Meta SRE: menos de 50% do
tempo.

**Error Budget** Diferença entre 100% e o SLO. Representa quanto um sistema pode
falhar sem violar o objetivo de confiabilidade. Usado para equilibrar velocidade
e estabilidade.

**Blameless Postmortem** Análise de incidente focada em aprendizado, não em
culpa. Prática fundamental para cultura de confiabilidade.

**Guardrails** Restrições automatizadas que impedem ações de alto risco.
Exemplo: impedir deploy em produção durante horário de pico.

### Conceitos de Observabilidade

**Cardinalidade** Número de valores únicos para uma métrica. Alta cardinalidade
aumenta custo e complexidade de armazenamento.

**Telemetry** Dados coletados sobre comportamento de sistemas: métricas, logs,
traces, eventos.

**Golden Signals** Quatro métricas críticas de saúde de serviço: Latency,
Traffic, Errors, Saturation.

**Distributed Tracing** Rastreamento de requisições através de múltiplos
serviços, essencial para arquiteturas de microserviços.

### Conceitos de Arquitetura

**Microserviço** Estilo arquitetural onde aplicação é composta de serviços
pequenos, independentes e especializados.

**Service Mesh** Camada de infraestrutura que gerencia comunicação entre
microserviços (ex: Istio, Linkerd).

**Circuit Breaker** Padrão que impede chamadas a serviços degradados, evitando
cascata de falhas.

**Canary Release** Técnica de deployment que libera nova versão para pequeno
subconjunto de usuários antes de rollout completo.

### Conceitos de Infraestrutura

**GitOps** Paradigma onde operações são definidas em repositório Git e
sincronizadas automaticamente com infraestrutura.

**Immutable Infrastructure** Prática de substituir componentes ao invés de
modificá-los, garantindo consistência e facilidade de rollback.

**Self-Healing** Capacidade de sistema detectar e corrigir automaticamente
falhas sem intervenção humana.

**Chaos Engineering** Prática de injetar falhas controladas para testar
resiliência de sistemas.

### Conceitos de Machine Learning

**Overfitting** Quando modelo ML aprende dados de treino mas não generaliza para
dados novos.

**Feature Engineering** Processo de criar e selecionar variáveis (features) para
treinamento de modelos ML.

**Inference** Processo de usar modelo treinado para fazer predições em dados
novos.

**Causalidade vs. Correlação** Correlação indica associação estatística;
causalidade indica que uma variável causa efeito em outra. Sistemas de RCA
buscam causalidade.

## Métricas DORA Detalhadas

### Deployment Frequency

| Nível  | Definição                                  | Características                      |
| ------ | ------------------------------------------ | ------------------------------------ |
| Elite  | On-demand (múltiplos deploys por dia)      | Deploys automatizados, feature flags |
| High   | Entre uma vez por dia e uma por semana     | CI/CD maduro, pouca burocracia       |
| Medium | Entre uma vez por semana e uma por mês     | Alguma automação, gates manuais      |
| Low    | Entre uma vez por mês e uma a cada 6 meses | Processo manual, muitos aprovadores  |

### Lead Time for Changes

| Nível  | Definição                 | Características                       |
| ------ | ------------------------- | ------------------------------------- |
| Elite  | Menos de uma hora         | Deploy contínuo, testes automatizados |
| High   | Entre um dia e uma semana | CI/CD funcional, testes integrados    |
| Medium | Entre uma semana e um mês | Testes parciais, aprovações manuais   |
| Low    | Entre um mês e 6 meses    | Processo waterfall, testes manuais    |

### Change Failure Rate

| Nível      | Percentual | Implicação                                  |
| ---------- | ---------- | ------------------------------------------- |
| Elite/High | 0-15%      | Qualidade embutida no processo              |
| Medium     | 16-30%     | Alguma dívida técnica, testes insuficientes |
| Low        | 31-45%+    | Processo quebrado, muitos hotfixes          |

### Time to Restore Service

| Nível  | Definição                 | Características                        |
| ------ | ------------------------- | -------------------------------------- |
| Elite  | Menos de uma hora         | Runbooks automatizados, rollback fácil |
| High   | Menos de um dia           | Boa observabilidade, equipe treinada   |
| Medium | Entre um dia e uma semana | Processo de incidente definido         |
| Low    | Mais de uma semana        | Caos, sem processo claro               |
